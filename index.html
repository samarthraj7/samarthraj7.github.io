Agricultural Yield Prediction with Satellite Data

Published at ICICC 2024, Springer

üéØ Situation: A Personal Problem

Remember my father's farming struggles? This was my chance to address it. Farmers in Karnataka (and across India) lack access to precise, location-specific crop forecasts. They rely on intuition and generic regional data, leading to:

  • Poor crop selection decisions
  • Suboptimal yield
  • Financial instability
The Challenge: Agricultural predictions in India are typically region-wide. But conditions vary dramatically even within a few kilometers. Could we provide square-kilometer level predictions?

üìã Task: Building a Hyper-Local Prediction System

My Role: Lead Researcher & Implementation Lead

Team: 4 members (I led the data science approach)

Goal 1:

Achieve predictions at 1km¬≤ resolution (unprecedented for our region)

Goal 2:

Use 12 years of historical data to capture seasonal patterns

Goal 3:

Make it practical - farmers should be able to use coordinates to get forecasts

‚ö° Action: The Technical Journey

1
Data Collection & Processing
  • Extracted 12 years of MODIS satellite imagery from NASA's EarthData
  • Calculated NDVI (Normalized Difference Vegetation Index):
    NDVI = (NIR - Red) / (NIR + Red)
  • Processed imagery for Karnataka region: extracted Red and NIR bands
  • Generated monthly averages to address seasonal variations
Challenge Solved: MODIS provides 30-day intervals - we needed monthly granularity for crop cycle alignment.
2
Model Architecture: Why LSTM?

Initial attempts with traditional ML models (Random Forest, XGBoost) showed limitations in capturing temporal dependencies. Agricultural patterns aren't just about current conditions - they depend on what happened in previous months.

LSTM (Long Short-Term Memory) Network Architecture:
  • 3 sequential LSTM layers (capturing long-term dependencies)
  • Time step: 12 months (one full crop cycle)
  • Dense output layer with MSE loss function
  • ADAM optimizer for efficient training
# LSTM equations for one time step:
i_t = œÉ(W_i ¬∑ [h_{t-1}, x_t] + b_i)  # Input gate
f_t = œÉ(W_f ¬∑ [h_{t-1}, x_t] + b_f)  # Forget gate
o_t = œÉ(W_o ¬∑ [h_{t-1}, x_t] + b_o)  # Output gate
C_t = f_t ¬∑ C_{t-1} + i_t ¬∑ tanh(W_c ¬∑ [h_{t-1}, x_t] + b_c)
h_t = o_t ¬∑ tanh(C_t)
3
Data Preprocessing & Feature Engineering
  • Min-Max Normalization: Scaled NDVI values to [0,1] range
    x' = (x - min(x)) / (max(x) - min(x))
  • Created 144 data points per location (12 years √ó 12 months)
  • Implemented 80-20 train-test split with temporal awareness (no data leakage from future)
  • Handled missing data from cloud cover using interpolation
4
Training & Optimization
  • Trained over 20 epochs with early stopping
  • Monitored training/validation loss curves for overfitting
  • Experimented with different learning rates and batch sizes
  • Cross-validated across different regions of Karnataka
5
Practical Application Layer

Building the model was half the battle. Making it usable was the other half.

  • Created crop-specific NDVI value tables for:
    • Rice (different growth stages: vegetative ‚Üí generative ‚Üí harvesting)
    • Corn/Jowar
    • Areca nut palm
    • Ragi
  • Developed 12-month crop cycle recommendations based on predicted NDVI
  • Example: For Yelandur Taluk, Chamarajanagar - recommended Rice (Jan-Aug) ‚Üí Corn (Sep-Dec) cycle
Problem-Solving Highlights
Challenge: Cloud cover obscuring satellite data
Solution: Implemented temporal interpolation and used composite imagery from multiple dates
Challenge: Different crops have vastly different NDVI patterns
Solution: Created crop-specific lookup tables validated against ground truth data
Challenge: Making predictions accessible to non-technical users
Solution: Simple coordinate-based interface with visual NDVI maps

Real-World Impact
  • Published at ICICC 2024 (Springer) - Research recognized internationally
  • Practical Application: Farmers can now input their location coordinates and get:
    • Predicted NDVI for next 12 months
    • Recommended crops for their specific location
    • Optimal planting and harvesting times
  • Personal Achievement: Solved a problem I witnessed firsthand growing up
Validation Approach

Tested predictions against actual crop yields from government agriculture department data for Karnataka. Our recommendations aligned with successful crop patterns in 85% of test cases.

üß† Learnings & Hindsight Considerations

What I'd Do Differently:
  • Incorporate Weather Data: While NDVI captures vegetation health, combining it with rainfall, temperature, and soil moisture data could improve predictions further
  • Real-time Updates: Current model uses monthly data. Implementing near-real-time predictions with weekly updates would be more practical
  • Multi-modal Approach: Exploring attention mechanisms and transformers (beyond LSTM) for potentially better temporal modeling
  • Ground Truth Collection: Partner with local agricultural departments for more extensive validation data
Key Takeaways:
  • Domain Knowledge Matters: Understanding crop cycles and agricultural practices was as important as ML expertise
  • Data Quality > Data Quantity: 12 years of clean, processed data beat having more years of noisy data
  • User-Centric Design: A brilliant model is useless if farmers can't use it. Simplicity in interface was crucial
  • Collaboration is Key: Working with my professor and team members with different expertise enriched the solution
Future Directions:
  • Expand to other states in India with different climate patterns
  • Integrate with mobile app for accessibility
  • Add economic modeling - not just "what to grow" but "what's most profitable"
  • Explore transfer learning to reduce data requirements for new regions

The Winning Edge: Cricket Scenario Recommender System

Published at WorldS4 London 2023, Springer

üéØ Situation: Strategy Meets AI

Remember how watching cricket sparked my journey? This project brought my childhood passion and AI expertise together.

The Problem: Professional cricket demands split-second strategic decisions - which shot to play, where to place fielders, which bowling type to use. Currently, this relies heavily on:
  • Coaches' experience and intuition
  • Basic statistical analysis (averages, strike rates)
  • Post-match video analysis (too late for in-game decisions)

What if we could provide real-time, AI-powered recommendations based on 500+ historical match scenarios? What if we could identify a batter's weaknesses or suggest optimal field placements during the match?

üìã Task: Building an Intelligent Recommender

My Role: Lead NLP Engineer & System Architect

Team: 4 members (I led the BERT implementation and overall system design)

Goal 1:

Analyze 500+ match scenarios to recommend alternate shots and field placements

Goal 2:

Achieve 85%+ tactical accuracy (validated against expert cricket analysts)

Goal 3:

Process ball-by-ball cricket commentary using NLP to extract actionable insights

Goal 4:

Make it context-aware - same bowler, different situations = different recommendations

What Made This Unique:

Unlike traditional cricket analytics that look at aggregate stats, we needed to understand natural language commentary and extract strategic patterns. This required advanced NLP, not just number crunching.

‚ö° Action: Building the Hybrid System

1
Dataset Creation: "StatHive"

Before building the recommender, we needed data. Not just stats, but contextual match situations.

  • Web Scraping: Collected ball-by-ball commentary from ESPNCricinfo and Cricbuzz
  • Structured Data Extraction: Parsed JSON data from Cricsheet for match details
  • Keyword Extraction using BERT: Identified specific shots, bowling types, and field placements from commentary
  • Preprocessing: Cleaned, standardized, and merged data
  • Feature Extraction: Created comprehensive features including:
    • Ball characteristics (line, length, pitch type)
    • Player information (batting style, bowling type)
    • Match context (runs required, wickets remaining, overs left)
    • Seasonal/venue conditions
Innovation: First-of-its-kind cricket dataset combining commentary NLP with structured match data - we called it "StatHive".
2
The Hybrid Architecture: ANN + BERT

Why hybrid? Because cricket decisions require both pattern recognition (neural networks) and language understanding (BERT).

Component 1: Artificial Neural Network (ANN)
  • Architecture:
    • Input layer: 3 nodes (ball characteristics)
    • Hidden layer: 5 nodes with ReLU activation
    • Output layer: 1 node with sigmoid activation
  • Purpose: Predict batting motive (attack vs. defend) based on:
    • Runs required
    • Balls remaining
    • Wickets down
# Sigmoid for output (probability of attack/defend):
Motive = M(Runs_required, Balls_remaining, Wickets_down)
where M is trained ANN predicting attack (1) or defend (0)
Component 2: BERT (Masked Language Model)
  • Pre-training: Fine-tuned BERT on cricket commentary corpus
  • Masked Prediction: Given context like "Right-arm leg break bowler to left-handed batter on [MASK] pitch"
  • Query Formation: Structured queries incorporating:
    • Batter type
    • Bowler type
    • Ball line and length
    • Ground type
    • Season/time of day
  • Output: Predicted alternate shots based on masked token filling
How They Work Together:
  1. ANN determines the motive (should batter attack or defend?)
  2. Based on motive, we filter our dataset:
    • If attacking ‚Üí look for scenarios where batter scored 4s/6s
    • If defending ‚Üí look for scenarios where batter played safe shots
  3. BERT takes this filtered context and recommends specific shots
  4. System also suggests field placements to counter these shots
3
Training & Optimization
ANN Training:
  • Dataset: Historical match scenarios with labeled motives
  • Loss function: Binary cross-entropy
  • Optimizer: ADAM
  • Validation: 80-20 train-test split
BERT Fine-tuning:
  • Base model: BERT-base-uncased
  • Fine-tuning corpus: 500+ match commentaries
  • Masked Language Modeling (MLM) objective
  • 15% token masking strategy:
    • 80% replaced with [MASK]
    • 10% replaced with random token
    • 10% unchanged
  • Training: 20 epochs with sparse categorical cross-entropy loss
4
Field Placement Recommendations

Shots are only half the story. We also needed to recommend defensive strategies for bowling teams.

  • Created template-based field placement repository
  • Mapped shot types to optimal field configurations
  • Considered factors:
    • Bowler-batter combination
    • Match situation (powerplay vs. death overs)
    • Batter's preferred scoring zones
    • Ground dimensions
  • Generated dynamic visualizations of recommended field placements
5
Real-time Query System

Built a query interface where you input:

  • "Right-handed batter facing right-arm fast bowler"
  • "Ball pitched full on leg stump"
  • "Required run rate: 8.5"

And get back:

  • Top 3 recommended alternate shots with success probability
  • Field placement diagram to counter these shots
  • Historical examples of similar scenarios
Problem-Solving Highlights
Challenge: Cricket commentary is highly contextual and uses jargon
Solution: Fine-tuned BERT specifically on cricket corpus rather than using generic pre-trained models
Challenge: Same ball characteristics, different situations = different optimal shots
Solution: Hybrid approach - ANN determines context (motive), BERT provides situation-specific recommendations
Challenge: Validating "tactical accuracy" - no ground truth labels
Solution: Collaborated with professional cricket coaches to validate recommendations against expert judgment
Challenge: Handling rare scenarios (e.g., super overs, unusual bowling styles)
Solution: Implemented fallback mechanisms and confidence scores for recommendations

What I'd Do Differently:
  • Video Analysis Integration: Combine commentary NLP with actual video footage analysis using computer vision
  • Player-Specific Fine-tuning: Train individual models for high-profile players where abundant data exists

  • More Granular Field Placements: Instead of template-based, use generative approaches for customized field configurations
Key Takeaways:
  • Domain Expertise is Crucial: My years playing cricket helped me understand nuances that pure data wouldn't reveal

  • Interpretability: Coaches wanted to know why the system recommended certain shots, not just what
